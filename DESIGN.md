## How the generator works

1. **Frame removal.** The source JPEG has a thin (~3px) gray rectangle around the
   artwork. The script insets `FRAME_INSET = 8px` on every edge before processing,
   then tight-trims, so no frame survives into the PNGs.
2. **Real alpha conversion.** This is a flat JPEG, so transparency is genuinely
   computed, not tagged. Per the brief, any pixel whose R, G and B are all above
   245 is flooded fully transparent. A short alpha *ramp* between min-channel 246
   (clear) and 220 (solid) is added below that hard cut — this removes the white
   halo that a pure hard threshold leaves on anti-aliased edges.
3. **Symbol / wordmark split.** Rather than hardcoding crop boxes, the script
   finds the **largest interior empty horizontal band** in the trimmed artwork
   (the gap between the cube and the wordmark) and splits there. This keeps working
   if the source logo is re-exported at a different size.
4. **Light recolor (pixel-based, chosen over CSS filter).** See below.

## Recolor approach — and why pixel-based won

The brief offered two options for the dark-surface variants: per-pixel recolor, or
a CSS `filter` fallback in the `<Logo />` component. **We use per-pixel recolor**
because it gives a noticeably cleaner, more controllable result here:

- A CSS `invert()/hue-rotate()` on the original logo would turn the navy into an
  arbitrary hue and can't independently preserve the cyan accent — the tie stripe
  and tagline would drift to the wrong color.
- Pixel recolor lets us map the two ink families independently: **navy → cream,
  cyan → brightened cyan**, with the cyan stripe kept intact.

**The non-obvious gotcha:** the brand navy `#152A37` is itself a *dark teal* — its
blue channel (0x37) already exceeds its red (0x15). So "blue is higher than red"
alone misclassifies the entire navy logo as cyan (the first run tinted everything
blue). The fix: the cyan accent is also **bright** (max channel ~190) while navy
ink is **dark** (max channel ~55). The classifier therefore requires *both* blue
dominance (`b - r > 28`) *and* brightness (`max(r,g,b) > 120`). Alpha is preserved
so edges stay smooth.

### Known limitation / future ask for the client

Because the navy-to-white anti-aliased edge ring passes through colors close to a
desaturated cyan, a faint cyan tint can appear on the brightest 1px edge ring of
the cream letters in the light variants. It is subtle at display sizes. For a
pixel-perfect result the client should eventually supply a **clean vector (SVG)**
of the logo; we would then recolor by swapping fill colors with zero edge
ambiguity. Until then, the generated PNGs are the canonical assets.

If a cleaner result is ever needed without an SVG, the documented fallback is a
CSS-filter `<Logo variant="light" />` (e.g. `brightness(0) invert(1)` for a flat
cream silhouette plus a separately-positioned cyan accent), accepting the loss of
the two-tone treatment.
